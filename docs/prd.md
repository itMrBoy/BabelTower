# BabelTower MVP PRD

## 1. 产品定位

BabelTower 是面向产品经理与前端 / Java 工程团队的 i18n 字典管理系统。它把散落在前端 JSON、浏览器扩展 JSON、Java properties 文件中的多语言文案沉淀为统一字典，支持多端复用、统一表达、减少重复翻译。

## 2. MVP 目标

- 支持录入、搜索、去重、冲突提示的全局中英文字典库。
- 支持单中文文件导入后自动回填英文、人工编辑、暂存、保存、同步字典、导出中英文双文件。
- 支持中文 + 英文双文件导入，额外展示字典命中值，并按“字典值优先、原英文兜底”导出。
- 支持按项目管理任务：同一项目的文件结构保持一致，最近一次任务可继续编辑，历史任务只读。
- 明确 JSON 与 properties 的统一中间结构，保证后续可以稳定互转。

## 3. 非目标

- MVP 不做复杂权限、组织架构、审批流、机器翻译自动生成。
- MVP 不保证逐字符保留原文件注释和空行，只保证 key 顺序、可编辑文案与导出结构稳定。
- MVP 不支持同一个任务内混合 JSON 与 properties 文件。
- MVP 不支持非字符串 leaf value 的翻译；解析时标记为 `UNSUPPORTED_VALUE` 并要求用户处理。

## 4. 角色与使用场景

| 角色 | 核心诉求 |
| --- | --- |
| 产品经理 | 查询已有中文 / 英文表达，导入文件后快速补齐英文，低成本统一文案 |
| 前端工程师 | 获得结构一致的 zh/en JSON 文件，减少手工查字典 |
| Java 工程师 | 获得结构一致的 zh/en properties 文件 |
| 项目负责人 | 按项目追踪最近一次可编辑任务和历史导出记录 |

## 5. 页面与功能

### 5.1 字典录入

- 用户输入中文、英文，可选备注、标签。
- 保存前必须执行冲突检测。
- 如果中文完全一致且英文一致：提示已存在，不重复写入。
- 如果中文完全一致但英文不同：弹出冲突弹窗，展示冲突个数、原因、已有英文、待写入英文，用户授权后才允许更新字典。
- 如果中文相似度 >= 90%：弹出冲突弹窗，展示相似中文、相似度、已有英文、待写入英文，默认阻止直接写入，允许用户选择忽略相似、编辑后重试或合并。

### 5.2 字典搜索

- 输入中文：返回字典中匹配的英文、相似中文建议、更新时间。
- 输入英文：返回字典中匹配的中文、相似英文建议、更新时间。
- 搜索结果突出显示完全命中、模糊命中和冲突风险。

### 5.3 翻译页面一：单中文文件

1. 用户导入中文文本文件，默认格式为 JSON，可切换 properties。
2. 系统解析为 Standard JSON，并生成 3 列可编辑预览表：
   - key 值列：不可编辑。
   - 左侧中文列：可编辑。
   - 右侧英文列：可编辑。
3. 系统按中文检索 Dictionary，命中后回填英文列。
4. 用户可以暂存，TaskSnapshot 保存当前预览数据，刷新或下次进入不丢失。
5. 用户保存时进行必填校验：key、中文、英文都必须存在。
6. 用户选择是否同步字典库：
   - 字典不存在：写入新 Dictionary。
   - 字典存在且英文不同：必须用户授权更新。
7. 用户导出完成后的 zh 与 en 两版文件。

### 5.4 翻译页面二：中文 + 英文双文件

1. 用户同时导入中文、英文两个文件。
2. 两个文件格式必须一致，都是 JSON 或都是 properties。
3. key 集合必须一致；缺失 key 进入校验错误列表，保存前必须修复。
4. 预览表包含 4 列：
   - key 值列：不可编辑。
   - 中文列：可编辑。
   - 原英文列：可编辑。
   - 字典填充列：可编辑，只填充字典命中值，未命中留空。
5. 保存必填校验调整为：同一个中文字段下，原英文列和字典填充列至少存在一个。
6. 导出的 en 文件使用整合值：字典填充列优先，字典值为空则使用原英文列。

### 5.5 任务清单

- 字典值全局通用，任务与文件归属于项目。
- 每个项目展示“最近一次可编辑任务”入口。
- 保存后的任务仍可作为最近任务继续编辑；当用户重新导入文件并创建新任务时，旧任务进入历史列表。
- 历史任务允许查看和导出，不允许编辑。
- 任务记录展示项目、任务名、格式、模式、更新时间、保存状态、冲突数。

## 6. 数据流向

强制数据流：

```text
Input (File) -> Parser -> Standard JSON -> Conflict Check -> Database
```

说明：

- Parser 只负责文件格式识别、语法解析、key/value 抽取。
- Standard JSON 是后续预览、冲突检测、保存、导出的唯一中间表达。
- Conflict Check 在任何 Dictionary 写入前执行。
- Database 只接收通过校验和冲突处理的数据；未解决冲突只写入 TaskSnapshot，不写入 Dictionary。

## 7. 验收标准

- 导入 JSON 或 properties 后，预览表 key 顺序与源文件一致。
- 单中文文件模式下，已有字典能自动回填英文列。
- 双文件模式下，字典填充列只填命中字典的值，未命中保持空。
- 保存时必填校验符合对应页面规则。
- 同步字典前能识别中文完全一致但英文不同的冲突。
- 相似中文达到 90% 及以上时能展示冲突提示和相似度。
- Dictionary 表对中文内容存在唯一约束。
- TaskSnapshot 能恢复用户最近一次暂存预览。
- 导出的 JSON / properties 文件格式与导入格式一致。

## 8. AI 时代 UI 风格候选

### A. Arc Console

- 关键词：玻璃拟态、低饱和渐变、侧边命令台、悬浮预览。
- 适合：产品经理高频查词与轻量编辑，界面克制、信息密度高。
- 建议：主色使用石墨黑 + 雾蓝 + 亮青状态色。

### B. Linear Lab

- 关键词：极简卡片、快捷键优先、时间线任务、细线分割。
- 适合：任务清单、历史版本、冲突处理等偏效率型场景。
- 建议：主色使用暖白 + 墨绿 + 橙色风险提示。

### C. Notion Atlas

- 关键词：文档式编辑、表格内联、空状态引导、轻 AI 提示气泡。
- 适合：字典录入、冲突修复和预览表手工调整。
- 建议：主色使用纸张米白 + 深蓝灰 + 柔和金色强调。
